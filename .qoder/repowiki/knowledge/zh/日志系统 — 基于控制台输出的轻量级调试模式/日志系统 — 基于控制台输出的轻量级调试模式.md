本仓库未引入任何结构化日志框架（如 SLF4J、Logback、Log4j2），后端与前端均采用最基础的输出方式，属于“无正式日志系统”的状态。

### 后端（Spring Boot）
- **依赖**：`pom.xml` 中仅引入 `spring-boot-starter-web`、MyBatis-Plus、SQLite JDBC、Lombok、Test，**未声明任何日志实现依赖**。
- **MyBatis SQL 日志**：通过 `application.yml` 的 `mybatis-plus.configuration.log-impl=org.apache.ibatis.logging.stdout.StdOutImpl` 将 SQL 语句直接打印到标准输出，便于开发调试。
- **业务代码中的日志**：
  - `GlobalExceptionHandler.handleException` 使用 `e.printStackTrace()` 输出异常堆栈；
  - `DatabaseInitializer.run` 使用 `System.out.println("数据库初始化完成")` 输出启动信息。
- **配置**：`application.yml` 中无 `logging.level`、`logging.file`、`logging.pattern` 等 Spring Boot 日志属性，也未提供 `logback-spring.xml`/`log4j2.xml` 等配置文件。
- **结论**：后端日志完全依赖 JVM 标准输出，无任何级别控制、文件滚动或结构化字段能力。

### 前端（Vue 3 + Vite）
- **错误输出**：在多个 `.vue` 组件中使用 `console.error(e)` 输出异常；用户交互错误通过 Element Plus 的 `ElMessage.error(...)` 以 UI 消息提示。
- **统一拦截器**：`src/api/request.js` 对请求失败进行 Promise.reject，但内部没有 `console.warn/info/debug` 等结构化调用。
- **结论**：前端仅使用浏览器原生 `console.*` 与 UI 提示，无集中式前端日志库（如 loglevel、pino-browser）。

### 开发者约定与建议
- 当前项目处于原型/演示阶段，采用控制台输出足以满足本地调试需求。
- 若后续需要生产可观测性，建议：
  - 后端引入 `spring-boot-starter-logging`（默认包含 Logback），并通过 `logging.level.*` 和 `logging.file.name` 管理级别与落盘；
  - 前端引入轻量日志库并在 `request.js` 中统一记录请求/响应上下文。
